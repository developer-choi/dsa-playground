# 약수와 배수 (Divisors and Multiples) - 최적화 풀이 해설

## 1. 개요
이 코드는 주어진 정수 배열에서 각 숫자에 대해 **"나의 약수이거나, 나의 배수인 수가 배열 내에 몇 개 존재하는가?"**를 구하는 문제입니다. 이중 반복문($O(N^2)$)을 사용하지 않고, **에라토스테네스의 체(Sieve of Eratosthenes)** 원리를 응용하여 시간 복잡도를 획기적으로 줄인 방식입니다.

---

## 2. 핵심 원리: "일석이조 (One Pass, Two Updates)"

이 알고리즘의 가장 큰 특징은 약수와 배수를 따로 구하지 않고, **반복문 한 번으로 동시에 처리**한다는 점입니다.

* **기존 방식:** 내 배수 찾고, 다시 내 약수 찾고 (두 번 일함)
* **이 코드의 방식:** **"내가 형(배수) 집에 놀러 가면, 형 입장에서는 동생(약수)이 온 것이다"**라는 논리를 사용합니다.

즉, `i`가 `j`의 약수라면, 반대로 `j`는 `i`의 배수입니다. 이 관계를 이용해 `i`가 `j`를 방문할 때 **두 사람의 카운트를 동시에 올려줍니다.**

---

## 3. 상세 로직 분석

### Step 1. 빈도수 저장 (Frequency Array)
가장 먼저 배열에 있는 **숫자들의 등장 횟수**를 셉니다.
* `freq[3] = 2`라면, 숫자 3이 배열에 2개 있다는 뜻입니다.
* 이렇게 해두면 배열 전체를 뒤질 필요 없이, 특정 숫자가 몇 개 있는지 즉시($O(1)$) 알 수 있습니다.

### Step 2. 체(Sieve)를 이용한 순회 (Core Logic)
1부터 최대값(`MAX`)까지 루프를 돌며 배수 관계를 확인합니다.

1.  **기준 숫자 (`i`):** 1부터 시작해 1씩 증가합니다.
2.  **타겟 숫자 (`j`):** `i`부터 시작해 **`i`만큼 점프**하며 커집니다. ($i, 2i, 3i, 4i \dots$)
    * 이렇게 점프하면 `j`는 무조건 `i`의 **배수**가 됩니다.
    * 반대로 `i`는 무조건 `j`의 **약수**가 됩니다.

### Step 3. 상호 카운트 업데이트 (Mutual Update)
`i`와 `j`가 만났을 때, 두 가지 경우로 나누어 점수를 매깁니다.

#### A. `i`와 `j`가 서로 다른 경우 (`i != j`)
* **`i` 입장에서:** `j`는 나의 **배수**입니다.
  👉 `j`가 등장한 횟수(`freq[j]`)만큼 내 점수(`res[i]`)를 올립니다.
* **`j` 입장에서:** `i`는 나의 **약수**입니다.
  👉 `i`가 등장한 횟수(`freq[i]`)만큼 내 점수(`res[j]`)를 올립니다.

#### B. `i`와 `j`가 같은 경우 (`i == j`)
* 자기 자신을 만난 경우입니다.
* 문제 조건에서 **"자기 자신을 제외한(`j != i`)"** 개수를 세라고 했습니다.
* 따라서 `freq[j]`에서 **나 자신(1명)을 뺀 나머지 개수**만 더해줍니다.

---

## 4. 성능 복잡도 분석 (Sieve 최적화 방식)

이 알고리즘의 최종 시간 복잡도는 다음과 같습니다.

$$O(N + M \log M)$$

여기서 변수의 의미는 다음과 같습니다.
* **$N$**: 입력 배열의 데이터 개수 (Length of Array)
* **$M$**: 입력된 숫자 중 최대값 (Max Value in Array)

---

### 💡 복잡도 유도 과정 (Step-by-Step)

전체 알고리즘은 크게 **3단계**로 나뉩니다. 각 단계별 비용을 합산하여 계산합니다.

#### 1단계: 빈도수 테이블 생성 (Frequency Array)
입력 배열 `arr`를 한 번 순회하면서 각 숫자가 몇 번 나왔는지 `freq` 배열에 기록합니다.
* **연산:** 데이터 개수($N$)만큼 반복
* **비용:** **$O(N)$**

#### 2단계: 배수/약수 관계 찾기 (Core Logic)
이 알고리즘의 핵심이자 시간이 가장 많이 소요되는 부분입니다.
이중 반복문처럼 보이지만, 실제로는 **조화 급수(Harmonic Series)** 형태를 띱니다.

* **로직:**
    * $i = 1$ 일 때: $1, 2, 3, \dots, M$ 방문 (총 $\frac{M}{1}$회)
    * $i = 2$ 일 때: $2, 4, 6, \dots, M$ 방문 (총 $\frac{M}{2}$회)
    * $i = 3$ 일 때: $3, 6, 9, \dots, M$ 방문 (총 $\frac{M}{3}$회)
    * ...
    * $i = M$ 일 때: $M$ 방문 (총 $\frac{M}{M} = 1$회)

* **총 연산 횟수 합계:**
  $$\frac{M}{1} + \frac{M}{2} + \frac{M}{3} + \dots + \frac{M}{M}$$

  위 식을 $M$으로 묶으면 다음과 같습니다.
  $$M \times \left( 1 + \frac{1}{2} + \frac{1}{3} + \dots + \frac{1}{M} \right)$$

* **수학적 근거 (조화 급수):**
  괄호 안의 식 $(1 + \frac{1}{2} + \dots + \frac{1}{M})$은 수학적으로 **$\ln M$ (자연로그)** 에 수렴한다는 것이 증명되어 있습니다.

* **비용:** **$O(M \log M)$**

#### 3단계: 결과 매핑 (Result Mapping)
계산된 `res` 배열의 값을 원래 입력 순서(`arr`)에 맞춰서 반환합니다.
* **연산:** 데이터 개수($N$)만큼 반복
* **비용:** **$O(N)$**

---

### 📊 최종 결과 합산

$$\text{Total Time} = O(N) + O(M \log M) + O(N)$$

$O(N)$은 중복되므로 하나로 합치고, 가장 지배적인 항(Dominant Term)을 남기면:

$$\therefore O(N + M \log M)$$

### 📝 결론
* 데이터 개수($N$)와 숫자의 크기($M$)가 둘 다 10만($10^5$) 수준일 때:
    * **$N$ (10만)** + **$M \log M$ (약 170만)** $\approx$ **180만 번 연산**
* 이는 일반적인 이중 반복문 $O(N^2)$의 **100억 번** 연산에 비해 획기적으로 빠르며, 1초 이내에 충분히 통과 가능한 효율입니다.
