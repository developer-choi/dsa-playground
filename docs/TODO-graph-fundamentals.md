# TODO: 그래프 기초 학습 (방향성 + 표현법)

작성 맥락: wf-4(디렉토리 경로) 풀이 분석 중, **그래프 방향/표현/visited 사용 기준**을 정식으로 안 배운 상태로 코테 풀고 있었음을 발견. 면접 후 정리해서 학습할 것.

## 핵심 깨달음 (잊지 마라)

> **DFS의 본질은 '노드를 한 번씩 방문하기'가 아니라, '방문할 때 무엇을 함께 들고 다니냐'다.**

- 도구(스택/재귀/큐)는 골격일 뿐. 차이 거의 없음
- **알고리즘의 90%는 "스택에 노드 + 필요한 상태(누적값/depth/좌표 등)를 튜플로 묶어 push하는 패턴"**
- 본인 wf-4 풀이가 O(N²) 됐던 이유: 스택에 노드 번호만 넣고, 부모 정보를 별도 자료구조로 관리해서 매번 복사함
- 최적 풀이는 `[node, 누적길이]` 튜플로 push → 분기에서 자동 복제 → O(N)

## 학습 항목

### 1. 그래프 방향
- [ ] 무방향 그래프 vs 단방향(방향) 그래프 — 차이와 의미
- [ ] 어떤 관계가 무방향이고 어떤 게 단방향인지 판별 기준
  - 친구/도로/결혼 = 무방향 (서로)
  - 팔로우/부모-자식/포함/의존성/일방통행 = 단방향
- [ ] 트리도 결국 그래프의 일종 (사이클 없는 연결 그래프)
- [ ] 사이클 / DAG / 트리 분류 이해

### 2. 그래프 표현
- [ ] **인접 리스트 (Adjacency List)** ← 실전 90%, 이거 먼저
  - `graph[node] = [neighbors]` 형태
  - 입력(간선 리스트)을 인접 리스트로 변환하는 코드 패턴 손에 익히기
- [ ] 인접 행렬 (Adjacency Matrix)
  - `matrix[i][j] = 1/0` — 노드 적고 빠른 lookup 필요할 때
- [ ] 간선 리스트 (Edge List)
  - `[[u, v], ...]` — 입력 형태로 자주 등장. 보통 인접 리스트로 변환

### 3. visited의 정확한 사용 기준
- [ ] visited를 **왜** 쓰는가? (사이클 방지 + 중복 방문 제거)
- [ ] **트리(단방향)**에서 visited 불필요한 이유
- [ ] **DAG**에서도 visited가 필요한 이유 (지수 폭발)
- [ ] **사이클 그래프**에서 visited 필수 (무한 루프)
- [ ] 자료구조 선택 (Set / boolean 배열 / 비트마스크)

판정 표:
| 그래프 종류 | visited 필요? |
|---|---|
| 트리 (단방향) | ❌ 불필요 |
| 트리 (무방향) | ✅ 필요 |
| DAG | ✅ 필요 |
| 사이클 있는 그래프 | ✅ 필수 |

### 4. DFS/BFS 응용 패턴
- [ ] DFS 재귀 vs 스택 구현 — 같은 골격, 다른 도구
- [ ] BFS 큐 (`arr.shift()` 함정 — `let head = 0; queue[head++]` 인덱스 방식)
- [ ] **스택/큐에 (노드, 상태) 묶어 push 패턴** ← 이게 핵심
- [ ] **누적 파라미터 패턴 (top-down)** — 부모 → 자식으로 누적값 전달
- [ ] **서브트리 결과 반환 패턴 (bottom-up)** — 자식 결과 합산
- [ ] 트리 지름 같은 양방향 패턴

## 영상/강의 다시 찾을 때 판별 기준

기존 코테 강의는 DFS를 "노드 번호만 스택에 넣는" 입문 형태로 가르치는 경우 많음. **다음 키워드 명시적으로 다루는 강의를 찾을 것:**

- "스택에 (노드, 상태) 튜플로 push하는 패턴"
- "누적 파라미터를 들고 다니는 DFS"
- "트리에서 visited가 불필요한 이유"
- "DAG에서 visited가 필요한 이유"

이 패턴을 안 가르치면 입문 수준 강의. 응용 못 함.
