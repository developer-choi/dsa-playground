문제 풀이 학습 플로우를 단계별로 진행하라.

## 컨벤션

- 파일명·카테고리: `.claude/contexts/convention.md` 참고
- 커밋 scope: 카테고리 폴더명과 동일 (예: `stack-queue`, `greedy`)

## 절차

### 1단계: 스캐폴딩
- `./scaffold.md`의 절차에 따라 풀이 파일과 테스트 파일 생성
- $ARGUMENTS로 전달된 URL 사용

### 2단계: 접근법 설계
- 사용자에게 안내: "풀이 접근법과 테스트 케이스를 한글로 작성해서 제출해주세요. 테스트 케이스는 테스트 파일에 주석으로 작성해주세요."
- 사용자가 "다 썼어", "작성했어" 등 완료를 알리면 풀이 파일과 테스트 파일을 직접 Read한다.
- **접근법 리뷰**: 풀이 파일의 한글 설명을 읽고 아래를 검증한다. 문제가 있으면 3단계로 넘어가지 않고 수정을 요청한다.
  - 자료구조에 무엇을 저장하는지 명시되어 있는가
  - 조건 분기마다 어떤 연산을 하는지 구체적으로 적혀 있는가 (예: "저장한다" X → "현재 인덱스 - pop된 인덱스를 answer에 저장한다" O)
  - 순회가 끝난 후 잔여 원소 처리가 명시되어 있는가
- **테스트케이스 리뷰**: 사용자가 작성한 주석 케이스를 검토하여 빠진 케이스(경계값, 동일값, 단일원소 등)가 있으면 지적한다. 문제가 없으면 바로 다음 단계로.
- 현재 상태 커밋
- 빠진 케이스가 있으면 보완하여 추가 커밋
- 테스트 파일의 한글 주석(`// ...`)을 `it(...)` 코드로 변환하고 커밋

#### 막혔을 때 프로토콜
사용자가 접근법을 못 잡거나 문제가 이해 안 된다고 하면, 코드 작성을 막고 아래 순서로 끌고 간다:

1. **예시 탐색** — 다른 입력 예시를 2-3개 직접 떠올려보게 하고 패턴을 말하게 한다
2. **자료구조 연결** — 배운 자료구조(스택, 큐, 이진탐색 등) 중 적용 가능한 게 있는지 묻는다
3. **특징 글로 쓰기** — 문제의 특징을 사람이 하는 행동처럼 한글로 풀어쓰게 유도한다

코드를 먼저 작성하려는 낌새가 보이면 즉시 제지: "예시 먼저요."

### 3단계: 풀이 작성
- 사용자에게 안내: "풀이를 작성해서 제출해주세요."
- 사용자가 "다 썼어", "작성했어" 등 완료를 알리면 풀이 파일을 직접 Read한다. "제출해주세요"라고 재요청하지 않는다.
- **버그 발견 시**: 수정 방법을 절대 알려주지 않는다. 버그를 드러내는 테스트케이스를 추가하고, 풀이 파일과 테스트 파일을 **함께** 커밋한다. 사용자가 스스로 수정하게 한다.
- 파일을 읽은 뒤 커밋

### 4단계: 검증
- `.claude/skills/review/SKILL.md`의 검증, 분석 절차를 따른다
- 개선점이 있으면 피드백 후 커밋
