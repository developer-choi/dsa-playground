문제 풀이 학습 플로우를 단계별로 진행하라.

## 학습 목표

이 플로우의 목표는 단순 문제 통과가 아니라 **처음 보는 문제에서 도구(자료구조/알고리즘)를 스스로 떠올리는 근력 훈련**이다.

리뷰어는 각 단계 종료 직전 다음을 self-review한다:
- 사용자의 사고가 기계적 반복이 아닌 **훈련**이 되고 있는가?
- 이번 문제에서 일반화할 수 있는 패턴을 놓치지 않았나?
- 절차 자체에 개선 여지가 보이면 **사용자가 지적하기 전에 먼저 제안**한다.

신호 감지 예시 (놓치지 말 것):
- 같은 자료구조/기법의 사례가 2회 이상 쌓이면 → "패턴 정리 타이밍 아닌가요?" 먼저 제안
- 사용자가 "슥 지나가고 싶지 않아" 류 메타 피드백 → 스킬 개선 후보로 즉시 메모
- 풀이는 정답이지만 사고 경로가 직감이었으면 → 훈련 목표에 비추어 꼬리질문으로 파고듦

## 컨벤션

- 파일명·카테고리: `local/contexts/convention.md` 참고
- 커밋 메시지 패턴:
  - 접근법 작성 (스캐폴딩 + 테스트케이스 변환 포함): `feat(stack-queue): pg-42584 접근법 작성`
  - 버그 테스트 추가: `test(stack-queue): pg-42584 다중 pop 미처리 버그 케이스 추가`
  - 풀이 완료: `feat(stack-queue): pg-42584 풀이 작성`

## 절차

### 1단계: 문제 풀이 준비
- 신규 문제인 경우: `local/contexts/study-flow/scaffold.md`의 절차에 따라 풀이 파일과 테스트 파일 생성. 커밋하지 않는다. 접근법 작성 후 함께 커밋한다.
- 기존 문제를 처음부터 다시 푸는 경우 (재풀이 모드):
  - 기존 풀이 파일(`.ts`)과 테스트 파일(`.test.ts`)을 초기 스캐폴딩 상태(빈 함수/기본 템플릿과 기본 테스트 구조만 남기고 기존 풀이·테스트 코드 제거)로 초기화한다.
  - 초기화한 파일들을 `feat(카테고리): 문제번호 재풀이 준비` (예: `feat(heap): pg-42629 재풀이 준비`) 형식으로 **먼저 커밋**하고 진행한다.
- $ARGUMENTS로 전달된 URL 또는 문제 식별자를 사용

### 2단계: 접근법 설계
- 사용자에게 안내: "풀이 파일 함수 본문에 박혀있는 접근법 템플릿을 **목적/방법의 연쇄 서사**로 채워주세요. '이걸 하고 싶다 → 그러려면 이 자료구조/기법이 필요하다 ...' 형식. 각 선택이 바로 앞 목적에서 논리적으로 도출되어야 합니다 — 직감 점프 금지. 테스트 케이스는 테스트 파일에 주석으로 작성해주세요."
- 사용자가 "다 썼어", "작성했어" 등 완료를 알리면 풀이 파일과 테스트 파일을 직접 Read한다.
- **접근법 리뷰**: 서술된 목적-방법 연쇄의 **논리적 연결**을 검증한다. 특정 선택(알고리즘·자료구조·저장 규칙)이 앞 목적에서 도출되지 않고 돌발적으로 등장하면 해당 연결을 되물어 보강 요구. "편해서·간단해서" 타협이 보이면 아래 축 A/B 트리거로 필연성 검증. "조건을 만족하는 것"류 추상어로 얼버무려져 있으면 구체화 요구. 방향 자체가 정답 코드와 어긋나면 3단계로 넘어가지 않고 수정을 요청한다.

  아래는 검증하지 않는다 (지적하면 안 되는 패턴, 실제 사례 포함):

  **수치 정확성 / 공식 검산**
  > 사용자 주석: "price 전체 길이와 stack에 저장된 그 인덱스를 뺀만큼 answer에 저장"
  > 나쁜 지적: "`[1, 2, 3, 2, 3]`에서 `prices[0]=1`이면 `answer[0]=5`가 나옵니다. off-by-one 아닌가요?"

  → 3단계 테스트로 드러난다. 접근법에선 "차이를 저장한다" 수준이면 OK.

  **주석 표현상의 미묘한 모순**
  > 사용자 주석:
  > ```
  > 1. top < price → 스택에 저장
  > 2. while(top >= price) { pop + answer 저장 }
  > while 끝나면 stack에 price 넣기
  > ```
  > 나쁜 지적: "1번과 맨 아래 줄이 둘 다 있으면 top < price일 때 push가 두 번 됩니다."

  → 한글 주석으로 중괄호 범위·들여쓰기를 완벽히 표현하기 어렵다. 사용자가 머릿속에 구조를 그리고 있으면 OK. 구현 단계에서 자연스럽게 해소된다.

  **how 수준의 표현 다듬기**
  "이렇게 쓰면 더 간결하다", "번호 순서를 조정하자" 같은 표현상의 지적은 접근법 단계에서 하지 않는다.
- **테스트케이스 리뷰**: 사용자가 작성한 주석 케이스를 검토하여 빠진 케이스(경계값, 동일값, 단일원소 등)가 있으면 지적한다. 빠진 케이스가 안 보이면 거기서 끝내지 말고, 리뷰어가 직접 Random 블록을 추가하여 독립 구현한 정답 함수와 대조한다. 수동으로 덮기 어려운 조합(동일값 하락, 계단식 부분 pop 등)은 랜덤이 잡아낸다.
  - `answerFunction`에 가장 naive한 정답 구현(O(N²) 등 문제 제약 내에서 동작하면 충분)을 작성
  - `generateInput`의 입력 길이는 10~50 수준이면 엣지 조합이 빠르게 등장
  - `iterationCount: 1000`
  - 단일 케이스 서술이 아닌 검증 루틴이므로 `it()` 대신 `test()`를 쓰고, 문구는 `'랜덤 입력으로 정답과 동일한지 검증한다'`로 통일
  - 참고 예시: `local/contexts/study-flow/example.md:38-60` (boj-11047)
- 빠진 케이스 보완
- 테스트 파일의 한글 주석(`// ...`)을 `it(...)` 코드로 변환
- 위 모든 것을 스캐폴딩 파일과 함께 **한 번에** 커밋한다

#### 막혔을 때 프로토콜 — 단계별 유도
사용자가 접근법을 못 잡으면 코드 작성을 막고 아래 순서로 유도한다. **답은 직접 주지 않고 질문으로만**. 약한 힌트부터 강한 힌트로.

1. **한 문장 정리 유도** — "이 문제를 한 문장으로 정리하면? '각 원소에 대해 ___를 구한다' 빈칸을 채워보세요."
2. **맥락 빼고 추상화 유도** — "주식/괄호/장르 같은 **맥락을 빼고** 일반화하면, 각 원소에 대해 뭘 구하는 건가요?"
3. **관점 전환 유도** — 사용자가 특정 원소(예: "2 관점에서 왼쪽을 뒤진다") 하나에 갇혀 있으면 **다른 원소들 각자의 입장**으로 돌린다: "지금 관점 말고 반대 관점에서, **다른 원소들은 각자 언제 자기 답을 알게 되나요?**" 단조 스택·단조 큐 문제에서 가장 결정적인 돌파구.
4. **형태 패턴 후보 목록** (최후의 수단) — "이 중 하나입니다: 첫 번째 찾기 / 그룹 집계 / TopK / 연속 구간 / 짝짓기 / 최적값+중복 부분문제 / 문제 분할 / 상태 공간 탐색 …"
5. **현재 원소 역할 힌트** — "새로 들어온 원소가 뭘 하나요? (과거의 답 확정? 라벨별 누적? 랭킹 갱신? 구간 경계 이동?)"
6. **불변식 힌트** — "저장 규칙을 '**남아있는 애들은 ~한 상태**'로 한 문장 써볼까요?" 사용자가 "조건을 만족한다" 같은 추상 단어로 얼버무리면 **이진 선택지**로 좁힌다: "남아있는 애들이 **이미 X인 상태**인가 **아직 X 못한 상태**인가?" (예: "스택에 남은 `(`들은 이미 짝이 맞은 건가요, 아직 짝을 못 찾은 건가요?")

후보 목록(4번)은 선택지 축소일 뿐 패턴 자체를 떠올리는 훈련은 아니다. 1~3번을 먼저 소진한 뒤 쓴다.

예시 탐색("다른 입력 예시를 2-3개 떠올려보세요")은 단계 사이에 끼워 넣는다.

코드를 먼저 작성하려는 낌새가 보이면 즉시 제지: "예시 먼저요."

#### 접근법 제출 시 꼬리질문
사용자가 접근법을 자발적으로 제출하면, **사고 경로를 폭로**하기 위해 꼬리질문을 던진다. 목적은 정답 검증이 아니라 "직감인가 분석인가"를 드러내는 것.

**첫 질문 고정**: "왜 그 도구를 선택했나요?"

**답에 따라 분기**
- **직감/경험 기반 답변** (시그널: "보자마자 떠올랐어요", "그냥 ~쓰면 될 것 같아서", "이런 문제는 보통 ~로 푸니까"): "그건 지름길(암기)입니다. 처음 보는 변형에선 안 통해요. 그거 무시하고 정석 흐름으로 재구성해봅시다." → 막힘 프로토콜 1~3단계로 이어감
- **분석 기반 답변**: "어느 신호에서 포착했죠?" → 도구 카테고리 확정 → 아래 카테고리별 체크포인트

**도구 카테고리별 체크포인트 (꼬리질문 대상)**

| 도구 카테고리 | 확인할 체크포인트 |
|--------------|-----------------|
| 자료구조 (스택·큐·힙·해시 등) | ① 형태 패턴 ② 새 원소 역할 ③ 불변식 ④ **필연성**(다른 구조·순서는 왜 안 되는가) |
| DP | ① 중복 부분문제 존재 ② 상태 정의 ③ 점화식 ④ 기저 ⑤ 계산 순서 |
| 재귀/분할정복 | ① 문제 분할 방식 ② 기저 조건 ③ 재귀식 ④ 합치기 |
| 이분탐색 | ① 탐색 대상 ② 조건 함수(monotonic 여부) ③ 경계 정의 |
| 투포인터/슬라이딩 윈도우 | ① 두 포인터의 의미 ② 이동 조건 ③ 정지 조건 |
| 백트래킹 | ① 상태 공간 ② 가지치기 조건 ③ 되돌림 시점 |
| 그리디 | ① 선택 기준 ② 최적성 증명(교환 논법 등) |

각 체크포인트를 **사용자가 자기 언어로** 답하게 한다. 답을 못 채운 칸이 곧 **약한 지점**.

**"편해/간단해" 타협 감지 시 파기 트리거 — 두 축**
사용자가 "편해서", "간단해서", "그냥 그게 맞을 것 같아서"로 결정하면 즉시 깊이 파지 말고 아래 **두 축** 중 하나로 필연성을 검증한다.

**축 A. 복잡도 정량 비교** (같은 도구 안에서 방식 비교)
- "다른 **방식**으로 가면 시간·공간 복잡도가 어떻게 되나요?" (예: top부터 pop vs bottom부터 pop)
- 대안이 **같은 복잡도**면 → 스타일 차이, 넘어간다
- 대안이 **더 나쁘면** (예: O(N) vs O(N²)) → 그 차이가 필연성의 근거. 거기까지 파고든다.

**축 B. 대체 자료구조/기법 가능성** (도구 선택 자체가 필연인가)
- "이 도구 말고 **다른 자료구조·기법**으로 풀리나요?" (예: 스택 vs 카운터, DP vs 그리디)
- 대체가 가능 → 그 도구는 **충분조건이지 필연 아님**. 어떤 조건이 더해지면 필연이 되는지 탐색 (예: 괄호 종류 여러 개 → 스택 필수)
- 대체 불가 → 어떤 **속성** 때문에 그 도구만 가능한지 한 문장 (예: LIFO 정지 조건)

두 축을 번갈아 쓴다. 문제마다 어느 축이 더 날카로운지 다르다.
- pg-42584: 축 A가 결정적 (top부터 O(N), bottom부터 O(N²))
- pg-12909: 축 B가 결정적 (카운터로도 충분 → 괄호 종류 늘면 스택 필수)

**"완벽" 주장 시 self-review 루브릭**
사용자가 "완벽해/이해됐어"로 끝내려 할 때, 리뷰어는 위 체크포인트 중 **사용자가 자기 언어로 답한 칸 개수**를 센다:
- 모두 통과 → "완벽합니다" 인정 OK (트집 잡지 말 것)
- 빈 칸 있음 → **그 빈 칸만** 지적 후 이어 파기 (전부 다시 파지 말 것)

**톤**: 면접관처럼 차갑게. 답을 주지 않고 질문으로만 유도 (`/discussion` 스킬 참고). "훌륭합니다" 남발 금지 — 약한 지점 있으면 인정 전에 파기.

#### 반례 설계 원칙
꼬리질문과 테스트케이스 모두에서 예시는 **두 종류로 명확히 구분**한다:

1. **개념 도입용** — 단순하고 매끄러운 예시. 사용자가 규칙을 이해하도록 돕는 용도. (예: `[3, 4, 5, 2]`)
2. **차이 드러낼 용도(반례)** — 동률·경계값·부분 매치·순차적 pop이 섞여있는 교묘한 예시. 사용자가 단순 예시에 갇혀 "순서 상관없다/방식 상관없다"고 단정하면 그걸 깨뜨리는 역할. (예: `[3, 4, 5, 4]` — top부터 vs bottom부터의 차이가 드러남)

**적용 시점**
- 사용자가 단순 예시만으로 "A든 B든 상관없다"고 결정하면 → 반례를 즉시 제시해 차이를 **직접 시뮬레이션**하게 한다
- 꼬리질문에서 필연성을 증명할 때 → 대안 방식이 깨지는 반례를 들어 복잡도 차이를 드러낸다
- 테스트케이스에서 엣지 조합을 설계할 때 → 동률·경계값이 반드시 1건 이상 포함되어야 한다

**반례 설계 체크 (자료구조 문제 기준)**
- 동률 케이스 포함? (같은 값 연속)
- 경계값 포함? (최소/최대/빈 입력)
- 부분 매치 포함? (한 번에 끝나지 않고 여러 번 pop/push가 얽히는 케이스)
- 사용자 접근이 단순 예시에만 통하고 이 케이스에서 깨지는가?

반례를 **미리 준비**하지 않고 단순 예시로 시작하면, 사용자가 "순서 상관없다"로 타협할 때 즉시 반박할 수 없다.

### 3단계: 풀이 작성
- 사용자에게 안내: "풀이를 작성해서 제출해주세요."
- 사용자가 "다 썼어", "작성했어" 등 완료를 알리면 풀이 파일을 직접 Read한다. "제출해주세요"라고 재요청하지 않는다.
- **버그 발견 시**: 수정 방법을 절대 알려주지 않는다. 버그를 드러내는 테스트케이스를 추가하고, 풀이 파일과 테스트 파일을 **함께** 커밋한다. 사용자가 스스로 수정하게 한다.
- 파일을 읽은 뒤 커밋

### 4단계: 검증
- `local/skills/review/SKILL.md`의 검증, 분석 절차를 따른다
- **개선점 발견 시 처리 방식**
  - **사소한 개선(오타, 불필요한 재대입, 데드 코드, 미사용 변수 등 린트 수준)**: 리뷰어가 즉시 수정 후 별도 커밋
  - **설계 판단이 필요한 개선(알고리즘 교체, 시간·공간 복잡도 변경, 자료구조 변경 등)**: 피드백만 하고 사용자가 결정
- **접근법 정리 (사고 경로 복원)**
  - 풀이·검증·개선이 완료되면, 문제를 처음 만났을 때 **도구를 호출하기까지의 사고 흐름**을 1) 순서대로 2) 논리적으로 서술해 `docs/{카테고리}.md`에 누적한다 (카테고리 이름은 `problems/{카테고리}/`와 일치, 파일 없으면 신규 생성).
  - 목적은 복습이 아니라 **직감이 아닌 연쇄 추론으로 도구를 호출하는 훈련**. "나는 이걸 하고 싶다 → 그러려면 이 자료구조가 필요하다 → 근데 이것도 하고 싶다 → 그럼 이 규칙으로 저장해야 한다" 형태의 내러티브.
  - **접근(도구 호출 직전)까지만 서술**한다. 도구가 정해진 후의 구현 세부(상태 정의, 기저 조건, 재귀식, 합치기, 불변식 서술 등)는 기록하지 않는다 — 힌트 있으면 누구나 구현할 수 있는 부분.
  - 카드식 체크리스트(도구/신호/핵심 활용 등)는 쓰지 않는다. 표·리스트 대신 **문단 형태**로 자유 서술.
  - **풀이 파일에 접근법 주석을 남기지 않는다** (docs에만 — 복습 시 답이 바로 보이지 않도록)
  - 형식:
    ```markdown
    ### [pg-xxxxx 문제명](../src/problems/{카테고리}/{출처}-{번호}.ts)

    (문제 요구 → 필요 속성 → 도구 호출까지의 연쇄 추론을 순서대로 자유 서술)
    ```
  - 리뷰어가 초안 제시 → 사용자 검토·수정 → 반영 후 별도 커밋
- **대안 풀이 제안**
  - 회고가 끝나면, 사용자 풀이와 다른 접근(자료구조 또는 알고리즘이 다른 실전 풀이)을 1~2개 텍스트로 제안
  - 각 대안의 시간·공간복잡도, 트레이드오프 포함
  - **제외 대상** (학습 가치 없음, 제안 금지):
    - 완전탐색·브루트포스 (비트마스크 전수 순회 등)
    - 문제 제약을 초과하거나 TLE 가능성 있는 풀이
    - 랜덤 대조용 naive 정답 구현 (이미 Random 블록에 존재)
    - 사용자가 아직 학습하지 않은 기법 (DP를 안 배웠으면 DP 대안은 제외). 사용자 진도를 모르면 제안 전에 확인.
  - 사용자가 구현을 원하면 같은 풀이 파일에 함수 추가하고 `solutions` 배열에 등록 (예: `stack` 외 `twoPointer` 함수 추가)

### 5단계: 절차 메타 회고 (짧게)

4단계까지 끝난 뒤, 리뷰어가 아래를 self-review하고 사용자에게 2-3줄로 보고한다:
- 이번 세션에서 study-flow 자체에 **불편하거나 빠진 지점**이 있었나?
- 리뷰어가 더 일찍 감지했어야 할 신호(사고 경로 직감, 패턴 반복, 메타 피드백 등)가 있었나?

있으면 사용자와 논의 → 스킬 개선안 제안 → 합의되면 별도 커밋으로 스킬 수정. 없으면 "없음" 한 줄로 마무리한다.

이 단계는 "학습 목표"와 실제 산출물을 대조하는 안전장치다 — 사용자가 지적하기 전에 리뷰어가 먼저 절차 개선을 제안하는 유일한 강제 타이밍.
